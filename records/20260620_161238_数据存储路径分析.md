## 二、逐模块详细分析

### 2.1 Shazam 音频指纹库

**存储路径**: `data/shazam_db.json`（进程内内存 + JSON 磁盘持久化）

**存储方式**:
- 主存储：进程内内存（`_MemDB` 线程安全单例）
- 持久化：JSON 文件 `data/shazam_db.json`，启动时加载、退出时自动保存
- 实现文件: `backend/core/shazam/database/in_memory.py`

**数据结构**:
```python
{
    "music_table": [       # 曲目表
        {"music_id": int, "music_name": str, "music_path": str}
    ],
    "finger_prints": {     # 指纹数据: music_id -> [(hash, offset), ...]
        int: [[str, int], ...]
    },
    "hash_index": {        # 哈希索引: hash -> [(music_id, offset), ...]
        str: [[int, int], ...]
    },
    "next_music_id": int
}
```

**路径解析**: `InMemoryDatabaseChecker.DEFAULT_STORAGE_FILENAME = "data/shazam_db.json"`，相对于项目根目录。

### 2.3 参考音频

**存储路径**: `data/ref/`

**用途**: 作为 Shazam 指纹匹配的参考音频，用于长音频分析和精确定位

现有文件:
- `asd_src_audio.wav` — 主参考音频
- `青藏高原片段_10s.wav` — 青藏高原片段
- `渡口片段10s.wav` — 渡口片段

**录入方式**: 通过 API `/api/reference-audio/upload` 上传后，系统自动提取 Shazam 指纹存入内存数据库

**注意**: 参考音频的指纹存在内存数据库中（持久化到 `data/shazam_db.json`），原始音频文件只在上传时用于提取指纹，后续 Shazam 匹配不依赖原始音频文件存在。

### 2.4 数据集（SPK 音频数据集）

**存储路径**: `data/spk/`

**目录结构**:
```
data/spk/
├── {category_name}/            # 以参考音频名称命名的类别目录
│   ├── train/
│   │   └── good/               # 训练集：正常音频片段 (.wav)
│   └── test/
│       ├── good/               # 测试集：正常音频片段 (.wav)
│       ├── anomaly/            # 测试集：异常音频片段 (.wav)
│       └── bad/                # 兼容旧版数据
├── split_log.jsonl             # 数据集划分操作日志
├── inference.json              # 推理配置
├── meta.json                   # 数据集元信息
├── test_ref/                   # 测试参考音频目录
└── *.mp3                       # 未切分的原始音频文件
```

**存储方式**: 文件系统目录结构，按 MVTec AD 标准格式组织

**标注流程**:
1. 用户上传音频 → `uploads/dataset_temp/`
2. 系统自动 Shazam 匹配 + 切分 → `uploads/dataset_slices/`
3. 用户标注 normal/anomaly → 复制到 `data/spk/{category}/train/good/` 或 `test/good/` 或 `test/anomaly/`
4. 清理临时文件

**数据集划分规则**: 训练集:测试集 ≈ 10:1（异常数据只能进测试集）

## 六、关键数据流

### 6.1 离线检测流程

```
用户上传音频 (.wav/.mp3)
  → uploads/{task_uuid}/              [临时存储]
  → Shazam 指纹提取 + 内存数据库匹配
  → 时频图生成 → output/slices/picture/  [持久化]
  → 音频切片 → output/slices/audio/      [持久化]
  → 异常检测推理 (GPU)
  → 热力图 → output/vis/                 [持久化]
  → 结果保存到 localStorage (前端)       [浏览器缓存]
  → 清理 uploads/{task_uuid}/           [删除临时文件]
```

### 6.2 数据集构建流程

```
用户上传音频 → uploads/dataset_temp/      [临时存储]
  → Shazam 自动匹配参考音频
  → 切分片段 → uploads/dataset_slices/    [临时存储]
  → 用户标注 normal/anomaly
  → 按 10:1 划分训练/测试集
  → 复制到 data/spk/{category}/train|test/good|anomaly/  [持久化]
  → 生成时频图 (.png) 与音频 (.wav) 同目录           [持久化]
  → 记录划分日志 → data/spk/split_log.jsonl           [持久化]
  → 清理临时文件                                      [删除]
```

### 6.3 Shazam 指纹库生命周期

```
服务启动
  → InMemoryDatabaseChecker.check_database()
  → 从 data/shazam_db.json 恢复数据到内存            [磁盘→内存]
  → 注册 atexit 保存回调

服务运行中
  → 所有指纹操作直接操作内存 (_MemDB 单例)
  → 每次写操作后自动 auto_save() → data/shazam_db.json

服务退出
  → atexit 触发 InMemoryDatabaseChecker.save_on_exit()
  → 内存数据序列化到 data/shazam_db.json              [内存→磁盘]
```

## 七、注意事项

1. **无硬编码路径**: 所有路径通过 `backend/core/_paths.py` 的 `PROJECT_ROOT` 动态解析，或从 `config.yaml` 读取，支持部署到任意路径。

2. **Shazam 无外部依赖**: 已从 MySQL 迁移到进程内内存存储，无需安装任何数据库服务。

3. **模型文件需单独下载**: `.pth/.pt` 等模型文件未纳入 Git，部署时需从 HuggingFace/PyTorch Hub 等源下载，或从已有环境复制。

4. **CUDA 依赖**: 算法推理依赖 CUDA GPU，但可通过配置 `device=cpu` 降级使用 CPU。

5. **前端存储**: 离线检测历史结果存储在浏览器 localStorage 中，服务端不保存检测历史记录（仅保存输出文件）。
